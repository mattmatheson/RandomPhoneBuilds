Syncing Claude Code Across Machines

A complete guide for cloning Matt's full Claude Code setup — skills, memory, MCP servers, hooks, and all — from the MacBook to the Windows PC and Mac Mini.

Pro tip: For a better voice, select all then use iOS Speak Selection (Settings > Accessibility > Spoken Content).

The Big Picture

Right now, your MacBook is the fully loaded machine — 21 active skills, 31 archived skills, 6 MCP servers, TTS hooks, 20+ custom commands, your entire Cortex vault, and the claude-brain sync repo. The goal is to replicate that environment on two other machines:

  • Windows PC — your main editing/gaming rig, different OS, different path conventions
  • Mac Mini — fresh macOS, same OS family, way simpler to clone

There's no official "Claude Sync" feature from Anthropic yet (there's a feature request at GitHub issue #22648, but no timeline). So we're building our own sync pipeline using your existing claude-brain GitHub repo as the central hub.

The Sync Strategy: Git + Symlinks

Your claude-brain repo at github.com/mattmatheson/claude-brain is already the right vehicle. The approach is dead simple:

  1. The claude-brain repo holds the canonical copy of everything sync-worthy
  2. Each machine clones the repo and symlinks files back into ~/.claude/
  3. When you make changes on any machine, commit + push. Pull on the others.

The repo already has bootstrap.sh and sync.sh scripts. We'll expand those to handle the full setup.

Part 1 Windows PC Setup

This is the harder one — different OS, different paths, some features that work differently. Let's break it down phase by phase.

Phase 1: Install Claude Code on Windows

Claude Code runs natively on Windows 10/11 now (no WSL required). The recommended install method:

# Option A: winget (preferred)
winget install Anthropic.ClaudeCode

# Option B: npm (if winget is being weird)
npm install -g @anthropic-ai/claude-code

You also need Git for Windows installed — Claude Code uses Git Bash internally even when you're running it from PowerShell or CMD.

winget install Git.Git

After install, open a terminal and run claude to authenticate. It'll walk you through the Anthropic login. On Windows, your credentials land in %USERPROFILE%\.claude\.credentials.json (not Keychain like on Mac).

Security note: On Windows, credentials are stored in a plaintext JSON file, not the system keychain. Never sync this file. Never commit it to Git.

Phase 2: Clone the Brain

# From PowerShell or Git Bash:
cd %USERPROFILE%
git clone https://github.com/mattmatheson/claude-brain.git

This gives you the full config repo at C:\Users\Matt\claude-brain\ (or whatever your Windows username is).

Phase 3: Symlink Into .claude

Windows supports symlinks but requires either Developer Mode enabled or running as admin. Turn on Developer Mode in Settings → For Developers → Developer Mode.

Then the bootstrap script creates the links. Here's the core logic for bootstrap-win.ps1:

# bootstrap-win.ps1 — Run as Admin or with Developer Mode on
$brain = "$env:USERPROFILE\claude-brain"
$claude = "$env:USERPROFILE\.claude"

# Back up existing settings if present
if (Test-Path "$claude\settings.json") {
    Copy-Item "$claude\settings.json" "$claude\settings.json.bak"
}

# Create symlinks for sync-worthy items
$items = @(
    "CLAUDE.md", "settings.json", "commands", "skills",
    "skills-archive", "agents", "hooks", "plugins", "plans"
)

foreach ($item in $items) {
    $target = "$brain\$item"
    $link = "$claude\$item"

    if (Test-Path $link) { Remove-Item $link -Recurse -Force }

    if (Test-Path $target -PathType Container) {
        New-Item -ItemType Junction -Path $link -Target $target
    } else {
        New-Item -ItemType SymbolicLink -Path $link -Target $target
    }
    Write-Host "Linked: $item"
}

Phase 4: Handle the Path Problem

This is the main gotcha. Claude Code stores project-specific memory using path-encoded directory names:

  • MacBook: ~/.claude/projects/-Users-matthewmatheson/memory/
  • Windows: ~/.claude/projects/-C-Users-Matt/memory/

So memory files from the MacBook won't automatically be found on Windows — the project path keys are different.

Solution: Cross-link memory

In the bootstrap script, we create a symlink that maps the Windows project path to the same memory folder:

# In bootstrap-win.ps1, after the main links:
$memSrc = "$brain\memory"
$winProject = "$claude\projects\-C-Users-Matt"

New-Item -ItemType Directory -Path $winProject -Force
New-Item -ItemType Junction -Path "$winProject\memory" -Target $memSrc

Now both machines read/write the same memory files regardless of OS path differences.

Phase 5: MCP Servers

Your current MCP servers from settings.json:

  • sequential-thinking — NPX, works cross-platform
  • memory (knowledge-graph) — NPX, works cross-platform but needs path adjustment for the .jsonl file
  • context7 — NPX, works cross-platform
  • chrome-devtools — needs Chrome installed
  • gmail-personal / gmail-business — need credential files in the right spot

Most of these are NPX-based, so they'll work as long as Node.js is installed. The Gmail MCP servers need their credential/config directories recreated on the Windows machine. The knowledge-graph path needs to be updated in settings.local.json (the machine-specific override file).

settings.local.json (Windows overrides)
{
  "env": {
    "CLAUDE_MEMORY_PATH": "C:\\Users\\Matt\\.claude\\memory\\knowledge-graph.jsonl"
  }
}

Phase 6: Skills & External Toolkits

Your skills are mostly symlinks pointing to external toolkit repos:

  • ~/tools/nano-banana-2/ — hosts 31+ skills
  • ~/tools/claude-video-toolkit/ — hosts ffmpeg, remotion, elevenlabs, etc.
  • ~/.agents/skills/ — hosts firecrawl

For the sync repo, we flatten these — copy the actual skill files into claude-brain/skills/ instead of storing symlinks (symlinks to local paths won't survive cross-machine sync). The bootstrap script on each machine then re-links from the repo.

Phase 7: Windows-Specific Adjustments

  • TTS Hook: The tts-stop.py hook uses macOS say command, which doesn't exist on Windows. We'll need a Windows equivalent using PowerShell's Add-Type -AssemblyName System.Speech or skip TTS on Windows.
  • Line endings: Set git config core.autocrlf true on Windows to handle LF/CRLF conversion.
  • Playwright: Works natively on Windows. Run npx playwright install after setup.
  • Shell hooks: Any bash-based hooks need PowerShell equivalents or Git Bash wrappers.

Windows: Quick-Start Checklist

  1. Install Git for Windows: winget install Git.Git
  2. Install Node.js: winget install OpenJS.NodeJS.LTS
  3. Install Claude Code: winget install Anthropic.ClaudeCode
  4. Run claude and authenticate
  5. Enable Developer Mode (Settings → For Developers)
  6. Clone claude-brain: git clone https://github.com/mattmatheson/claude-brain.git
  7. Run bootstrap-win.ps1 as admin
  8. Install Playwright browsers: npx playwright install
  9. Set up Gmail MCP credentials (manual — copy from MacBook)
  10. Test: run claude and verify skills load, memory works, MCP connects

Part 2 Mac Mini Setup

# Install Claude Code

# Authenticate

Use the same username (matthewmatheson) on the Mac Mini. This means all your path-encoded project directories in ~/.claude/projects/ will match exactly. If you already set up a different username, you can still make it work with symlinks, but matching usernames eliminates that headache entirely.

Phase 3: What bootstrap.sh Does

The bootstrap script for macOS is clean — just symlinks:

#!/bin/bash
BRAIN="$HOME/claude-brain"
CLAUDE="$HOME/.claude"

# Backup existing
[ -f "$CLAUDE/settings.json" ] && cp "$CLAUDE/settings.json" "$CLAUDE/settings.json.bak"

# Symlink sync-worthy items
for item in CLAUDE.md settings.json commands skills skills-archive agents hooks plugins plans; do
    target="$BRAIN/$item"
    link="$CLAUDE/$item"
    [ -e "$link" ] && rm -rf "$link"
    ln -sf "$target" "$link"
    echo "Linked: $item"
done

# Memory symlink
mkdir -p "$CLAUDE/projects/-Users-matthewmatheson"
rm -rf "$CLAUDE/projects/-Users-matthewmatheson/memory"
ln -sf "$BRAIN/memory" "$CLAUDE/projects/-Users-matthewmatheson/memory"
echo "Linked: memory"

echo "Done. Run 'claude' to verify."

Ongoing Sync Workflow

Once all three machines are set up, keeping them in sync is straightforward:

Daily Workflow

Before starting work on any machine:

cd ~/claude-brain && git pull

After making changes (new skills, updated memory, changed settings):

cd ~/claude-brain && git add -A && git commit -m "sync: description" && git push

The sync.sh Script

A one-liner that pulls latest and verifies symlinks are intact:

#!/bin/bash
cd "$HOME/claude-brain"
git pull --rebase
./bootstrap.sh  # re-verify symlinks
echo "Synced at $(date)"

Conflict Resolution

If you edit memory or settings on two machines before syncing, you'll get merge conflicts. For .md files, these are easy to resolve — just pick the version you want or merge manually. For settings.json, be more careful and diff the changes.

What About Conversation History?

Conversation history (history.jsonl) and sessions are intentionally not synced. They're large, machine-specific, and not useful across devices. What IS synced is your memory — the distilled context that Claude loads every conversation. So when you start a fresh session on the Windows PC, Claude won't remember the exact conversation from the MacBook, but it will know who you are, your preferences, your projects, and your feedback. That's the stuff that actually matters.

Community Sync Tools Worth Knowing

If the git + symlinks approach feels too manual, the community has built some tools:

  • memoir (memoir.sh) — CLI that auto-detects your Claude config, backs it up, and restores it on another machine. Handles cross-platform path remapping automatically.
  • chezmoi + age — Gold standard for dotfile management. Encrypts sensitive files before they hit the repo. More setup, but more robust.
  • dotclaude — Syncs via private Git repo with intelligent JSON merging for settings files.
  • CCMS — Lightweight bash script using rsync over SSH. Good if your machines are on the same network (like MacBook ↔ Mac Mini on Tailscale).

For your setup, the git + symlinks approach is the right call — you already have the repo, and it keeps you in control of exactly what syncs.
